쿠버네티스 기반 GitHub Enterprise 펌웨어 CI/CD

Atlassian Bamboo + Bitbucket + 커스텀 Spring Boot 릴리스 앱을 GitHub 네이티브 CI/CD 플랫폼으로 대체합니다. 이 플랫폼은 GitHub Enterprise Server (GHES), GitHub Actions, 쿠버네티스 기반 Actions Runner Controller (ARC), Apache Kafka 메시지 백본, 전용 서버의 릴리스 오케스트레이터 GitHub App, 그리고 릴리스, 결함, 부서 간 업무를 Atlassian Jira로 미러링하는 Jira Bridge Service로 구성됩니다. 300명 이상의 개발자, 하루 5,000건 이상의 빌드, 하드웨어 인더루프 펌웨어 테스트, 그리고 모든 변경사항이 Jira 티켓으로 추적되는 워크플로우를 지원하도록 설계되었습니다.

온프레미스 / 사내망 개발자 300명 이상 일일 빌드 5,000건 이상 하드웨어 인더루프 Jira 통합 7년 릴리스 보관

러너 풀 분리

CI / CD / HW는 서로 다른 K8s 노드 풀에서 물리적으로 분리된 ARC RunnerScaleSet으로 운영됩니다. CI가 CD 자원을 고갈시킬 수 없습니다.

워크플로우가 테스트 트리거

CD/HW GitHub Actions 워크플로우가 인증된 HTTP를 통해 각 커스텀 테스트 시스템을 명시적으로 트리거합니다. 테스트 시스템은 Kafka를 통해 결과를 다시 게시합니다—불투명한 결합 없음.

Fail-safe CD 게이팅

Quality Gate Service가 Check Run을 게시하고, Environment 보호 규칙이 이를 요구합니다. 상태 없음 = 릴리스 차단.

Jira ↔ GitHub 브리지

모든 릴리스는 두 번 게시됩니다: GitHub Release와 Jira fixVersion. Bridge Service가 QGS 실패와 HW 결함으로부터 Jira 결함을 자동 생성합니다 — 커밋 + 테스트 핑거프린트로 중복 제거.

Jira 키 필수

조직 수준 필수 워크플로우가 PR 제목 또는 커밋이 유효한 Jira 이슈(FW-1234)를 참조하지 않으면 거부합니다. 브랜치 보호가 강제하며, Smart Commits가 티켓 전환을 자동으로 처리합니다.

영구 결과 저장

펌웨어 바이너리는 Object Lock과 함께 7년간 MinIO/Ceph에 저장되며, 메타데이터와 Jira 링크는 PostgreSQL, 메트릭은 InfluxDB + Grafana에 저장됩니다.

Kafka 백본

Apache Kafka가 프로듀서와 컨슈머를 분리합니다. 테스트 시스템, 러너, 오케스트레이터, Jira Bridge, 분석 모두가 재생 및 버스트 흡수 기능을 갖춘 버전 관리 토픽을 통해 통신합니다.

전용 서버의 오케스트레이터

릴리스 오케스트레이터 GitHub App은 전용 VM에서 실행됩니다(K8s 외부). 러너 클러스터 문제가 릴리스 제어 플레인을 중단시킬 수 없습니다.

상위 수준 아키텍처

주요 시스템과 통신 방식. GHES는 코드, 빌드, 바이너리 릴리스의 단일 진실 공급원이며, Jira는 부서 간 업무의 기록 시스템입니다 — 이슈, 태스크, 결함, 외부 부서가 보는 릴리스 기록을 관리합니다. Jira Bridge Service가 Kafka와 Jira REST API + 웹훅을 통해 양쪽을 동기화합니다. ARC가 쿠버네티스에서 러너를 확장하고, CD는 GitHub Environment 보호 규칙을 통해 Quality Gate Service에 의해 게이팅됩니다.

컴포넌트 설계

GHES — 기록 시스템

GitHub Enterprise Server는 모든 저장소, 워크플로우, 릴리스 이벤트를 호스팅합니다. 핵심적으로 활용하는 기능:

  • 공유 CI 패턴을 위한 재사용 가능한 워크플로우 및 복합 액션.
  • 기본 CD 차단 메커니즘인 필수 검토자와 보호 규칙을 갖춘 Environment.
  • 조직 수준 필수 워크플로우 — 모든 저장소가 의무적인 정책/보안 작업을 실행.
  • 릴리스 오케스트레이터를 위한 저장소별로 범위가 지정된 설치 토큰을 갖춘 GitHub Apps.
  • 키 없는 러너-서비스 인증(S3, Postgres, HW 예약)을 위한 OIDC 토큰.

러너 — 쿠버네티스 기반 Actions Runner Controller

ARC는 GitHub Actions 셀프 호스팅 러너를 위한 정식 쿠버네티스 오퍼레이터입니다. 러너를 임시 파드로 자동 확장합니다—작업당 새 파드, 완료 후 삭제. 악용이 자리잡을 영구 상태가 없습니다.

세 가지 독립적인 RunnerScaleSet:

풀레이블네임스페이스트리거리소스 프로필
CIci, linux-x64runners-ci-<team>PR + 기능 브랜치 push팀별 ResourceQuota
CDcd, releaserunners-cdrelease 및 deployment 이벤트만예약 용량, 선점 불가
HWhw-bench-<id>, firmware-<product>runners-hw벤치 레이블을 요청하는 워크플로우K8s nodeSelector + taint/toleration

커스텀 러너 이미지는 매일 밤 빌드되고, cosign으로 서명되며, 내부 레지스트리에서 가져오고, 승인된 툴체인만 포함합니다. 새 도구를 추가하려면 개발자가 runner-image 저장소에 PR을 제출해야 하며, 런타임에 설치할 수 없습니다.

릴리스 오케스트레이터 — 전용 서버 (Spring Boot 대체)

전용 서버에서 호스팅되는 GitHub App — Spring Boot 릴리스 앱을 실행하던 동일한 VM 또는 새로운 동급 VM. 명시적으로 K8s 러너 클러스터 내부가 아닙니다.

K8s 외부에 위치하는 이유:

  • Blast Radius 격리. 클러스터 업그레이드, ARC 버그, 네트워킹 문제가 릴리스 제어 플레인도 중단시킬 수 없습니다.
  • 안정적인 네트워크 ID. GHES 웹훅과 오퍼레이터 접근을 위한 고정 호스트명/IP — 파드 재시작으로 인한 혼란 없음.
  • Lift-and-Shift 경로. 레거시 호스트의 기존 OS 구성, 모니터링 에이전트, 백업 정책 재사용.

Kafka 토픽(release.lifecycle, deployment.events)에 프로듀스/컨슘하여 수평 확장이 가능하고 분석 컨슈머로부터 분리됩니다. 주요 책임:

  1. release.published 및 deployment.created 웹훅 수신.
  2. PostgreSQL에 commit SHA, 버전, 대상 HW, 요청자를 포함한 firmware_release 행 생성.
  3. Kafka에 release.lifecycle 이벤트 게시.
  4. production-firmware Environment에 GitHub Deployment를 열어 보호 규칙 트리거.
  5. Kafka의 deployment.events(CD 러너가 내보냄)와 CI/QGS의 Check Run 소비.
  6. 성공 시 S3 아티팩트가 Object Lock(compliance-7y)으로 태그되었는지 확인.

Bamboo에서 불투명한 Python 스크립트로 인코딩되었던 오케스트레이션 로직이 이제 개발자가 읽을 수 있는 투명하고 버전 관리된 YAML 워크플로우로 저장됩니다.

Quality Gate Service (CD 차단)

테스트 결과를 GitHub Check Run으로 변환합니다. 차단은 GitHub 자체에 의해 강제됩니다 — QGS가 다운되거나 Kafka가 정체되면 체크가 녹색으로 변하지 않아 게이트가 닫힌 상태를 유지합니다(fail-safe).

  1. CD/HW GitHub Actions 워크플로우가 릴리스 ID, commit SHA, 대상 HW, 콜백 상관 ID와 함께 인증된 HTTP(예: POST https://firmware-validation.internal/runs)를 통해 각 관련 커스텀 테스트 시스템을 명시적으로 트리거합니다.
  2. 테스트 시스템은 비동기적으로 실행되고, 완료 시 상관 ID를 담아 Kafka 토픽 test.results에 결과를 게시합니다.
  3. QGS가 test.results를 소비하고, release-policies 저장소의 버전화된 YAML에 있는 제품별 릴리스 기준에 따라 평가합니다.
  4. QGS가 success | failure로 qgs/release-criteria 체크를 게시합니다.
  5. production-firmware Environment가 해당 체크를 요구합니다 — 상태 없음 = 차단.

왜 HTTP 웹훅 대신 Kafka에서 소비하나요? 하루 5,000건 이상의 빌드는 버스트성 결과 볼륨을 의미합니다. Kafka가 테스트 시스템과 QGS를 분리하고, 버스트를 흡수하며, QGS 레플리카를 독립적으로 확장하고, 테스트 시스템이 새로운 컨슈머를 알 필요 없이 분석/감사에 동일한 스트림의 복사본을 제공합니다.

Jira Bridge Service — 부서 간 기록 시스템

K8s 네임스페이스 services-integration에 위치한 stateless 서비스로, Atlassian Jira Data Center를 GHES, QGS, 릴리스 오케스트레이터와 동기화합니다. GHES는 코드, 빌드, 바이너리 릴리스를 소유하고; Jira는 부서 간 대화를 소유합니다 — 이슈, 태스크, 결함, 고객 가시 릴리스 기록. Bridge는 둘 사이의 접점이며, Jira REST API와 통신할 수 있는 유일한 컴포넌트입니다.

왜 직접 호출 대신 Bridge인가?

  • 하나의 클라이언트, 하나의 rate-limit 예산. Jira REST API는 사용자별 동시성 제한이 있습니다. 다른 모든 서비스는 Kafka로 통신하고, Bridge만 Jira와 통신합니다 — 폭주하는 러너로 인해 자체 Jira에 실수로 DoS를 일으킬 수 없습니다.
  • Jira 장애 시 버퍼링. Jira가 유지보수로 다운되면 jira.commands가 Kafka에 쌓입니다. Jira가 복구되면 Bridge가 토픽을 순서대로 소진합니다. 어떤 릴리스 이벤트도 손실되지 않으며, 파이프라인의 어떤 것도 Jira를 기다리며 차단되지 않습니다.
  • 설계상 idempotent. 각 명령은 자연 키(commit SHA, release tag, test fingerprint)를 가집니다. Bridge는 Postgres에 jira_link 테이블을 유지하여 재생이 중복 티켓을 생성하지 않도록 합니다.
  • 단일 감사 추적. 모든 Jira 변경은 audit 항목이기도 합니다; Jira 관리자가 나중에 티켓을 편집하더라도 SIEM은 Bridge가 Jira에 한 작업을 볼 수 있습니다.

Outbound 흐름 (Kafka → Jira)

트리거 토픽Jira 동작
release.lifecycle (published)fixVersion을 Released로 표시; 생성된 릴리스 노트 첨부; 모든 연결된 티켓에 "Released in 1.2.3 → fw-1.2.3.bin" 코멘트; 부서 간 가시성을 위한 RELEASE 프로젝트 기록 생성.
deployment.events연결된 티켓에 배포 상태 / 대상 HW / 아티팩트 URL 코멘트.
jira.commands (create_defect)Bug 티켓 생성 — 단, 먼저 (test_name, commit_sha, bench_id) 핑거프린트로 jira_link를 조회. 열린 결함이 있으면 중복을 만들지 않고 새 실패를 코멘트로 추가.
jira.commands (transition)티켓을 워크플로우를 통해 전환(예: PR 열림 시 "In Progress", CI 통과 시 "In Review", 머지 시 "Done").

Inbound 흐름 (Jira → Kafka)

Jira는 모든 이슈 업데이트에 대해 Bridge로 웹훅을 보냅니다. Bridge가 필터링, 정규화, 관련 항목을 jira.events에 다시 게시하여, 각 서비스가 자체 Jira 웹훅 구독 없이도 반응할 수 있게 합니다.

  • 릴리스 요청: PM이 티켓을 "Approved for Release"로 전환하면 jira.events{type:release_request}가 생성됩니다. 릴리스 오케스트레이터가 이를 소비하고 요청된 fixVersion과 티켓 목록으로 펌웨어 저장소에 repository_dispatch를 발생시킵니다.
  • 결함 종료: QA가 결함 티켓을 종료하면 Bridge가 jira_link 행을 업데이트하여, 동일한 핑거프린트를 가진 다음 테스트 실패가 종료된 티켓을 재오픈하지 않고 새 티켓을 생성하도록 합니다.
  • Smart Commits: FW-1234 #close fixed null deref 같은 커밋은 Bridge에 의해 Jira 전환으로 변환되며, Atlassian의 Smart Commit 문법을 미러링합니다.

Jira 키 필수 강제

조직 수준 필수 워크플로우 — policy/jira-key.yml — 가 모든 push와 PR에 실행됩니다. PR 제목과 커밋 메시지에서 [A-Z]+-\d+를 grep하고, Jira REST API를 호출하여 키가 허용 목록 프로젝트에 존재하는지 확인하고, policy/jira-key Check Run을 게시합니다. main의 브랜치 보호가 해당 체크를 요구하므로, 실제 Jira 티켓 없이는 커밋을 머지할 수 없습니다. Bridge는 해결된 commit_sha → issue_keys 매핑을 jira_link에 기록하며, 이는 모든 다운스트림 "연결된 티켓에 코멘트" 단계가 읽는 데이터입니다.

양방향 진실, 단방향 권한. 코드는 GHES에서 권위적이고, 상태 / 우선순위 / 일정은 Jira에서 권위적입니다. Bridge는 사람이 편집한 Jira 필드를 절대 덮어쓰지 않으며(대신 코멘트를 추가), GHES에 커밋이나 태그를 push하지 않습니다. 각 측에 분리된 필드의 소유권을 부여함으로써 설계상 충돌을 방지합니다.

하드웨어 인더루프 러너

Bamboo의 "HW에 고정된 에이전트"가 커스텀 K8s 노드 레이블과 내부 HW 예약 서비스를 갖춘 ARC로 매핑됩니다:

  • 각 벤치 연결 노드는 firmware.corp/bench=bench-A-04 같은 label과 taint를 가지므로 HW 워크로드만 해당 노드에 배치됩니다.
  • 워크플로우는 runs-on: [self-hosted, hw-bench, firmware-productX]로 선택합니다.
  • 예약 서비스가 희소한 벤치에 대한 접근을 순차적으로 제어합니다 — 작업이 hwres acquire / release를 호출합니다. 원격 측정이 Grafana에 피드됩니다.
  • 참조: 오픈소스 LabGrid가 처음부터 작성하는 것보다 통합할 수 있는 검증된 옵션입니다.

다이어그램: Jira 통합 흐름

Jira는 다른 부서들이 사용하는 시스템입니다 — PM, QA, 하드웨어 엔지니어, 고객 지원 모두가 거기서 티켓을 읽고 씁니다. 중요한 두 가지 경로: (A) GitHub 시작 — 엔지니어링이 출시하면 Bridge가 결과를 Jira로 미러링하여 다른 부서가 알 수 있게 합니다; 그리고 (B) Jira 시작 — PM이 Jira에서 핫픽스를 승인하면 Bridge가 그들을 대신해 GitHub 릴리스 워크플로우를 트리거합니다. 두 경로 모두 동일한 Kafka 토픽과 동일한 jira_link Postgres 테이블을 사용하므로, GitHub에서 시작하지 않은 릴리스도 Jira에 도달하고 그 반대도 마찬가지입니다.

리소스 거버넌스 — 오용 방지

여섯 가지 독립 레이어가 러너를 CI/CD 외 컴퓨팅에 사용하는 것을 방지합니다("빌드 팜의 암호화폐 채굴자" 위험):

  1. 임시 파드 러너 — 작업 후 영구 상태가 없습니다.
  2. 강화된 러너 이미지 — 승인된 툴체인만; 런타임에 apt install 불가; cosign으로 서명.
  3. 러너 네임스페이스의 NetworkPolicy — Egress 허용 목록은 GHES, 내부 레지스트리, MinIO, 승인된 서비스만. 퍼블릭 인터넷 차단.
  4. OPA Gatekeeper Admission Control — 러너 프로파일을 벗어나는 파드 거부.
  5. 필수 워크플로우 — 모든 실행이 안티패턴(긴 sleep, 의심스러운 바이너리)에서 실행을 실패시키는 정책 작업으로 시작합니다.
  6. Grafana의 Chargeback 대시보드 — 팀별 가시성이 자기 규제를 유도합니다.

CD 결과 미보관MinIO/Ceph Object Lock 7년 + 캡처용 NFS + 메타데이터용 PostgreSQL; 필수 복합 액션이 업로드 강제
기준 미충족 시 CD 차단Environment 보호가 qgs/release-criteria 체크 요구; fail-safe
HW 고정 실행레이블+테인트 K8s 노드의 ARC 러너 + HW 예약 서비스
개발자 유연성누구나 워크플로우 작성; 제약은 조직 필수 워크플로우 + 허용된 러너 레이블로 적용
온프레미스 / 사내망GHES, ARC, MinIO, Postgres, Grafana, QGS, Jira Bridge, HW 예약 모두 내부 K8s에 위치; Jira Data Center도 이미 온프레미스
다른 부서가 릴리스 가시성 필요(Jira)Bridge가 모든 릴리스를 Jira fixVersion으로 미러링, 연결된 티켓에 코멘트, 릴리스마다 RELEASE 프로젝트 기록 생성
결함은 Jira에 도달해야 함QGS 게이트 실패와 HW 벤치 결함이 jira.commands를 게시; Bridge가 Bug 티켓 생성, 커밋 + 테스트 핑거프린트로 중복 제거
부서 간 릴리스 요청Jira 티켓 전환 → 웹훅 → Bridge → repository_dispatch가 GitHub 릴리스 워크플로우 트리거
모든 변경이 Jira 티켓으로 추적조직 필수 워크플로우 policy/jira-key가 유효한 Jira 키 없는 PR 거부; Bridge가 다운스트림 조회를 위해 jira_link 채움

마이그레이션 계획

약 6~9개월에 걸친 단계적 롤아웃. 빅뱅 방식은 사용하지 마세요.

Phase 0 — 기초

4~6주. GHES, ARC, 파일럿 팀용 CI 풀, 러너 이미지 파이프라인, MinIO 버킷, Grafana 대시보드, 재사용 가능한 워크플로우 라이브러리 구축.

Phase 1 — CI 동등성

6~8주. 제품 라인 하나의 CI 마이그레이션. Bamboo Python 스크립트를 투명한 재사용 가능한 워크플로우로 분해. 두 릴리스 사이클 동안 병렬 운영.

Phase 2 — CD + 오케스트레이터

8~10주. 릴리스 오케스트레이터 GitHub App, QGS, HW 풀 구축. 제품 라인 하나의 릴리스 흐름 마이그레이션. Postgres 스키마 유지; 새 오케스트레이터를 연결.

Phase 2.5 — Jira Bridge

4~6주, Phase 2 후반과 병렬. K8s에 Jira Bridge 구축, jira.commands/jira.events 토픽 연결, Jira 웹훅을 Bridge에 등록. policy/jira-key 필수 워크플로우를 shadow mode(경고만)로 2주간 배포 후 강제로 전환. 지난 6개월간 머지된 PR에서 jira_link 백필.

Phase 3 — 규모 확장

순차적. 나머지 팀을 한 번에 하나씩 온보딩, 각 전환은 커스텀 테스트 시스템이 QGS에 연결되고 Jira 프로젝트가 Bridge 허용 목록에 추가되는 것으로 게이팅.

Phase 4 — 폐기

마지막 제품 라인이 이전을 완료하면 Bamboo와 Spring Boot 앱 종료. 이전에 Bamboo 웹훅으로 트리거되던 Jira 자동화 규칙은 Bridge로 재지정.

리스크 및 완화

리스크가능성영향완화
불투명한 Bamboo Python 스크립트가 동작을 숨김높음높음Phase 1 감사: 모든 스크립트를 재구현 전에 분해하고 문서화
커스텀 테스트 시스템의 형식이 다양함중간중간QGS가 안정적인 JSON 스키마 노출; 각 시스템이 얇은 어댑터 작성

GHES 중단으로 빌드 중단낮음높음HA 복제 + DR 런북; 릴리스 오케스트레이터가 재시도를 대기열에 추가
개발자가 필수 워크플로우를 우회낮음중간조직 수준 필수 워크플로우는 저장소 관리자도 우회 불가; 감사 로그 알림

Jira 장애로 릴리스 파이프라인 중단중간낮음jira.commands가 Kafka에 버퍼링(30일 보존); GitHub 릴리스는 차단 없이 진행; Jira 복구 시 Bridge가 백로그 처리
릴리스 폭주 시 Jira REST API rate limit 도달중간중간Bridge가 token-bucket 리미터와 프로젝트별 배칭을 갖춘 단일 Jira 클라이언트; jira.commands를 프로젝트 키로 파티션하여 병렬-제한 처리량
재생 또는 flapping 테스트로 인한 Jira 결함 중복중간낮음Bridge가 jira_link에서 (test_name, commit_sha, bench_id) 핑거프린트로 dedup; 재생성 대신 코멘트
Jira 웹훅 손실 또는 재생낮음중간야간 정합성 작업이 Jira fixVersion + 티켓 상태를 다시 읽고 보상 jira.events를 내보냄
개발자가 Jira 키 없이 커밋하여 PR 차단중간낮음스타터 저장소 템플릿의 pre-commit 훅 + IDE 플러그인이 로컬에서 경고; policy/jira-key 필수 워크플로우는 강제 전 2주간 warn-only 모드로 롤아웃
