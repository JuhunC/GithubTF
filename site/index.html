Firmware CI/CD on GitHub Enterprise

Replacing Atlassian Bamboo + Bitbucket + a custom Spring Boot release app with a GitHub-native CI/CD platform built on GitHub Enterprise Server (GHES), GitHub Actions, the Actions Runner Controller (ARC) on Kubernetes, an Apache Kafka message backbone, a Release Orchestrator GitHub App on a dedicated server, and a Jira Bridge Service that mirrors releases, defects, and cross-department work into Atlassian Jira. Designed for 300+ developers, 5,000+ builds/day, hardware-in-the-loop firmware testing, and a workflow where every change traces back to a Jira ticket.

On-prem / private network 300+ developers 5,000+ builds/day HW-in-the-loop Jira-integrated 7-year release retention

Separate runner pools

CI / CD / HW are physically separated ARC RunnerScaleSets on different K8s node pools — CI cannot starve CD.

Workflows trigger the tests

The CD/HW GitHub Actions workflow explicitly triggers each custom test system via authenticated HTTP. Test systems publish results back through Kafka — no opaque coupling.

Fail-safe CD gating

The Quality Gate Service publishes a Check Run; the Environment protection rule requires it. No status = release blocked.

Jira ↔ GitHub bridge

Every release is published twice: as a GitHub Release and as a Jira fixVersion. The Bridge Service auto-creates Jira defects from QGS failures and HW faults — deduplicated by commit + test fingerprint.

Required Jira key

An org-level Required Workflow rejects PRs whose title or commits don't reference a valid Jira issue (FW-1234). Branch protection enforces it; Smart Commits drive ticket transitions automatically.

Permanent results

Firmware binaries land in MinIO/Ceph with Object Lock for 7 years; metadata + Jira links in PostgreSQL; metrics in InfluxDB + Grafana.

Kafka backbone

Apache Kafka decouples producers from consumers: test systems, runners, Orchestrator, Jira Bridge, and analytics all communicate through versioned topics with replay and burst absorption.

Orchestrator on its own server

The Release Orchestrator GitHub App runs on a dedicated VM (replacing the Spring Boot host) — outside K8s — so a runner-cluster issue cannot bring down the release control plane.

Component design

GHES — system of record

GitHub Enterprise Server hosts every repo, workflow, and release event. The key features we lean on:

  • Reusable workflows & composite actions for shared CI patterns.
  • Environments with required reviewers and protection rules — the primary CD blocking mechanism.
  • Required workflows at the org level — every repo runs our mandatory policy/security jobs.
  • GitHub Apps with installation tokens scoped per repo for the Release Orchestrator.
  • OIDC tokens for keyless runner-to-service auth (S3, Postgres, HW reservation).

Runners — Actions Runner Controller on Kubernetes

ARC is the canonical Kubernetes operator for GitHub Actions self-hosted runners. It autoscales runners as ephemeral pods — fresh pod per job, destroyed afterward. No persistent state for misuse to take root in.

Custom runner images are built nightly, signed with cosign, pulled from an internal registry, and contain only approved toolchains. To add a new tool, a developer raises a PR against the runner-image repo — they cannot install at runtime.

Release Orchestrator — dedicated server (replaces Spring Boot)

A GitHub App hosted on a dedicated server — the same VM that previously ran the Spring Boot release app, or a fresh equivalent. Explicitly not inside the K8s runner cluster.

Why it lives outside Kubernetes:

  • Blast-radius isolation. A cluster upgrade, ARC bug, or networking issue cannot also take down the release control plane.
  • Stable network identity. Fixed hostname/IP for GHES webhooks and operator access — no pod-restart churn.
  • Lift-and-shift path. Reuses existing OS configuration, monitoring agents, and backup policy of the legacy host.

It produces to and consumes from Kafka (topics release.lifecycle and deployment.events) so it scales horizontally and decouples from analytics consumers. Responsibilities:

  1. Listen for release.published and deployment.created webhooks.
  2. Create a firmware_release row in PostgreSQL with commit SHA, version, target HW, requester.
  3. Publish a release.lifecycle event to Kafka.
  4. Open a GitHub Deployment on the production-firmware Environment, triggering protection rules.
  5. Consume deployment.events from Kafka (emitted by CD runners) plus check runs from CI/QGS.
  6. On success, confirm S3 artifacts are tagged for object lock (compliance-7y).

The orchestration logic that Bamboo encoded in opaque Python scripts now lives as transparent, version-controlled YAML workflows developers can read.

Quality Gate Service (CD blocker)

Translates test results into GitHub check runs. Blocking is enforced by GitHub itself — if QGS is down or Kafka is congested, the check never turns green and the gate stays closed (fail-safe).

  1. The CD/HW GitHub Actions workflow explicitly triggers each relevant custom test system via authenticated HTTP (e.g., POST https://firmware-validation.internal/runs) with the release ID, commit SHA, target HW, and a callback correlation ID.
  2. Test systems run asynchronously and publish results to Kafka topic test.results when complete, carrying the correlation ID.
  3. QGS consumes test.results and evaluates against per-product release criteria (versioned YAML in a release-policies repo).
  4. QGS publishes the qgs/release-criteria check with success | failure.
  5. The production-firmware Environment requires that check — no status = blocked.

Why pull from Kafka instead of HTTP webhooks? 5,000+ builds/day means bursty result volume. Kafka decouples test systems from QGS, absorbs bursts, scales QGS replicas independently, and gives analytics/audit a copy of the same stream — without test systems needing to know about new consumers.

Jira Bridge Service — cross-department system of record

A stateless service in K8s namespace services-integration that keeps Atlassian Jira Data Center in sync with GHES, QGS, and the Release Orchestrator. GHES owns code, builds, and binary releases; Jira owns the cross-department conversation — issues, tasks, defects, customer-visible release records. The Bridge is the seam between them, and the only component allowed to talk to the Jira REST API.

Why a bridge instead of direct calls?

  • One client, one rate-limit budget. Jira's REST API has per-user concurrency caps. Every other service speaks Kafka; only the Bridge speaks Jira — so we never accidentally DoS our own Jira from a runaway runner.
  • Buffering on Jira outage. If Jira is down for maintenance, jira.commands piles up in Kafka. When Jira comes back, the Bridge drains the topic in order. No release event is ever lost; nothing in the pipeline is blocked waiting on Jira.
  • Idempotent by design. Each command carries a natural key (commit SHA, release tag, test fingerprint). The Bridge keeps a jira_link table in Postgres so a replay never creates duplicate tickets.
  • Single audit trail. Every Jira mutation is also an entry on audit; the SIEM sees what the Bridge did to Jira even if a Jira admin later edits the ticket.

Inbound flows (Jira → Kafka)

Jira sends webhooks to the Bridge for every issue update. The Bridge filters, normalizes, and re-publishes the relevant ones to jira.events so any service can react without each one needing its own Jira webhook subscription.

  • Release request: a PM transitioning a ticket to "Approved for Release" produces jira.events{type:release_request}. The Release Orchestrator consumes it and fires repository_dispatch on the firmware repo with the requested fixVersion and ticket list.
  • Defect closed: when QA closes a defect ticket, the Bridge updates the jira_link row so the next test failure with the same fingerprint opens a fresh ticket instead of reopening a closed one.
  • Smart Commits: commits like FW-1234 #close fixed null deref get translated into Jira transitions by Bridge, mirroring Atlassian's own Smart Commit grammar.

Required-Jira-key enforcement

An org-level Required Workflow — policy/jira-key.yml — runs on every push and PR. It greps the PR title and commit messages for [A-Z]+-\d+, calls the Jira REST API to confirm the key exists in an allow-listed project, and posts a policy/jira-key Check Run. Branch protection on main requires that check, so a commit cannot be merged without a real Jira ticket attached. The Bridge writes the resolved commit_sha → issue_keys mapping into jira_link, which is what every downstream "comment on linked tickets" step reads.

Two-way truth, one direction of authority. Code is authoritative in GHES; status / priority / scheduling are authoritative in Jira. The Bridge never overwrites a Jira field a human edited (it appends a comment instead), and never pushes commits or tags into GHES. Conflict-by-design is avoided by giving each side ownership of disjoint fields.

HW-in-the-loop runners

Bamboo's "agents pinned to HW" maps onto ARC with custom K8s node labels and an internal HW Reservation Service:

  • Each bench-attached node carries a label like firmware.corp/bench=bench-A-04 and a taint, so only HW workloads land there.
  • Workflows opt in with runs-on: [self-hosted, hw-bench, firmware-productX].
  • The reservation service serializes access to scarce benches — jobs call hwres acquire / release. Telemetry feeds Grafana.
  • Reference: open-source LabGrid is a battle-tested option to integrate with rather than write from scratch.

Diagram: Jira integration flow

Jira is the system other departments live in — PMs, QA, hardware engineers, and customer support all read and write tickets there. Two paths matter: (A) GitHub-initiated — engineering ships, the Bridge mirrors the result into Jira so other departments learn about it; and (B) Jira-initiated — a PM approves a hotfix in Jira and the Bridge fires the GitHub release workflow on their behalf. Both paths use the same Kafka topics and the same jira_link Postgres table, so a release reaches Jira even when it didn't originate in GitHub, and vice versa.

Failure modes. Bridge down: jira.commands backs up in Kafka; the GitHub release still completes; Jira is updated when the Bridge returns. Jira down: Bridge marks commands "in flight," retries with exponential backoff, and emits an audit entry; the Orchestrator does not block the GitHub release on Jira availability. Webhook lost: the Bridge runs a nightly reconciliation job that re-reads Jira fixVersions and ticket states and emits compensating jira.events for anything missed.

Resource governance — preventing misuse

Six independent layers prevent runners from being used for non-CI/CD compute (the "crypto miner on the build farm" risk):

  1. Ephemeral pod runners — no persistent state survives a job.
  2. Hardened runner image — only approved toolchains; no apt install at runtime; signed with cosign.
  3. NetworkPolicy on runner namespaces — egress allowlist only to GHES, internal registry, MinIO, and approved services. Public internet blocked.
  4. OPA Gatekeeper admission control — denies pods escaping the runner profile.
  5. Required workflows — every run starts with a policy job that fails the run on anti-patterns (long sleep, suspicious binaries).
  6. Chargeback dashboards in Grafana — per-team visibility drives self-regulation.

Migration plan

Staged rollout over ~6–9 months. Do not big-bang.

Phase 0 — Foundation

4–6 weeks. Stand up GHES, ARC, CI pool for one pilot team, runner image pipeline, MinIO buckets, Grafana dashboards, reusable workflows library.

Phase 1 — CI parity

6–8 weeks. Migrate one product line's CI. Decompose Bamboo Python scripts into transparent reusable workflows. Run both in parallel for two release cycles.

Phase 2 — CD + Orchestrator

8–10 weeks. Stand up Release Orchestrator GitHub App, QGS, HW Pool. Migrate one product line's release flow. Keep the Postgres schema; point the new Orchestrator at it.

Phase 2.5 — Jira Bridge

4–6 weeks, parallel to late Phase 2. Stand up Jira Bridge in K8s, wire jira.commands/jira.events topics, register Jira webhook to Bridge. Ship the policy/jira-key Required Workflow in shadow mode (warn-only) for two weeks before flipping to enforce. Backfill jira_link from the last 6 months of merged PRs.

Phase 3 — Scale-out

Rolling. Onboard remaining teams one at a time, each cutover gated by their custom test systems being wired into QGS and their Jira project being added to the Bridge allow-list.

Phase 4 — Decommission

Retire Bamboo and the Spring Boot app once the last product line has moved. Any Jira automation rules previously triggered by Bamboo webhooks are repointed at the Bridge.

Risks & mitigations

RiskLikelihoodImpactMitigation
Opaque Bamboo Python scripts hide behaviorHighHighPhase 1 audit: every script decomposed and documented before re-implementation
Custom test systems' formats varyMedMedQGS exposes a stable JSON schema; each system writes a thin adapter
HW bench availability becomes the bottleneckMedHighReservation service emits queue-depth metrics → capacity decisions
GHES outage halts buildsLowHighHA replication + DR runbook; release Orchestrator queues retries
Developers route around Required WorkflowsLowMedOrg-level required workflows cannot be bypassed by repo admins; audit-log alerting
Storage cost growth (7y retention)HighLowTiered: hot S3 → cold object store after 1y; large captures to NFS not S3
Jira outage breaks the release pipelineMedLowjira.commands buffers in Kafka (30d retention); GitHub release proceeds unblocked; Bridge drains backlog when Jira returns
Jira REST API rate limits hit during release stormMedMedBridge is the single Jira client with a token-bucket limiter and per-project batching; jira.commands partitioned by project key for parallel-but-bounded throughput
Duplicate Jira defects from replay or flapping testsMedLowBridge dedup by (test_name, commit_sha, bench_id) fingerprint in jira_link; comments instead of recreating
Jira webhook lost or replayedLowMedNightly reconciliation job re-reads Jira fixVersion + ticket states and emits compensating jira.events
Developers commit without Jira keys, blocking PRsMedLowPre-commit hook in starter repo template + IDE plugin warns locally; policy/jira-key Required Workflow rolls out in warn-only mode for 2 weeks before enforcement
