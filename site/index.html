Firmware CI/CD on GitHub Enterprise

Replacing Atlassian Bamboo + Bitbucket + a custom Spring Boot release app with a GitHub-native CI/CD platform built on GitHub Enterprise Server (GHES), GitHub Actions, the Actions Runner Controller (ARC) on Kubernetes, an Apache Kafka message backbone, and a Release Orchestrator GitHub App hosted on a dedicated server — designed for 300+ developers, 5,000+ builds/day, with hardware-in-the-loop firmware testing.

On-prem / private network 300+ developers 5,000+ builds/day HW-in-the-loop 7-year release retention

Separate runner pools

CI / CD / HW are physically separated ARC RunnerScaleSets on different K8s node pools — CI cannot starve CD.

Workflows trigger the tests

The CD/HW GitHub Actions workflow explicitly triggers each custom test system via authenticated HTTP. Test systems publish results back through Kafka — no opaque coupling.

Fail-safe CD gating

The Quality Gate Service publishes a check run; the Environment protection rule requires it. No status = release blocked.

Permanent results

Firmware binaries land in MinIO/Ceph with object lock for 7 years; metadata in PostgreSQL; metrics in InfluxDB + Grafana.

Kafka backbone

Apache Kafka decouples producers from consumers: test systems, runners, Orchestrator, and analytics all communicate through versioned topics with replay and burst absorption.

Orchestrator on its own server

The Release Orchestrator GitHub App runs on a dedicated VM (replacing the Spring Boot host) — outside K8s — so a runner-cluster issue cannot bring down the release control plane.

Component design

GHES — system of record

GitHub Enterprise Server hosts every repo, workflow, and release event. The key features we lean on:

  • Reusable workflows & composite actions for shared CI patterns.
  • Environments with required reviewers and protection rules — the primary CD blocking mechanism.
  • Required workflows at the org level — every repo runs our mandatory policy/security jobs.
  • GitHub Apps with installation tokens scoped per repo for the Release Orchestrator.
  • OIDC tokens for keyless runner-to-service auth (S3, Postgres, HW reservation).

Runners — Actions Runner Controller on Kubernetes

ARC is the canonical Kubernetes operator for GitHub Actions self-hosted runners. It autoscales runners as ephemeral pods — fresh pod per job, destroyed afterward. No persistent state for misuse to take root in.

Three independent RunnerScaleSets:

Custom runner images are built nightly, signed with cosign, pulled from an internal registry, and contain only approved toolchains. To add a new tool, a developer raises a PR against the runner-image repo — they cannot install at runtime.

Release Orchestrator — dedicated server (replaces Spring Boot)

A GitHub App hosted on a dedicated server — the same VM that previously ran the Spring Boot release app, or a fresh equivalent. Explicitly not inside the K8s runner cluster.

Why it lives outside Kubernetes:

  • Blast-radius isolation. A cluster upgrade, ARC bug, or networking issue cannot also take down the release control plane.
  • Stable network identity. Fixed hostname/IP for GHES webhooks and operator access — no pod-restart churn.
  • Lift-and-shift path. Reuses existing OS configuration, monitoring agents, and backup policy of the legacy host.

It produces to and consumes from Kafka (topics release.lifecycle and deployment.events) so it scales horizontally and decouples from analytics consumers. Responsibilities:

  1. Listen for release.published and deployment.created webhooks.
  2. Create a firmware_release row in PostgreSQL with commit SHA, version, target HW, requester.
  3. Publish a release.lifecycle event to Kafka.
  4. Open a GitHub Deployment on the production-firmware Environment, triggering protection rules.
  5. Consume deployment.events from Kafka (emitted by CD runners) plus check runs from CI/QGS.
  6. On success, confirm S3 artifacts are tagged for object lock (compliance-7y).

The orchestration logic that Bamboo encoded in opaque Python scripts now lives as transparent, version-controlled YAML workflows developers can read.

Quality Gate Service (CD blocker)

Translates test results into GitHub check runs. Blocking is enforced by GitHub itself — if QGS is down or Kafka is congested, the check never turns green and the gate stays closed (fail-safe).

  1. The CD/HW GitHub Actions workflow explicitly triggers each relevant custom test system via authenticated HTTP (e.g., POST https://firmware-validation.internal/runs) with the release ID, commit SHA, target HW, and a callback correlation ID.
  2. Test systems run asynchronously and publish results to Kafka topic test.results when complete, carrying the correlation ID.
  3. QGS consumes test.results and evaluates against per-product release criteria (versioned YAML in a release-policies repo).
  4. QGS publishes the qgs/release-criteria check with success | failure.
  5. The production-firmware Environment requires that check — no status = blocked.

Why pull from Kafka instead of HTTP webhooks? 5,000+ builds/day means bursty result volume. Kafka decouples test systems from QGS, absorbs bursts, scales QGS replicas independently, and gives analytics/audit a copy of the same stream — without test systems needing to know about new consumers.

HW-in-the-loop runners

Bamboo's "agents pinned to HW" maps onto ARC with custom K8s node labels and an internal HW Reservation Service:

  • Each bench-attached node carries a label like firmware.corp/bench=bench-A-04 and a taint, so only HW workloads land there.
  • Workflows opt in with runs-on: [self-hosted, hw-bench, firmware-productX].
  • The reservation service serializes access to scarce benches — jobs call hwres acquire / release. Telemetry feeds Grafana.
  • Reference: open-source LabGrid is a battle-tested option to integrate with rather than write from scratch.

QGS uses idempotent processing so a replay never produces a duplicate check run. A wire-compatible alternative such as Redpanda is acceptable and lighter to operate.

Diagram: HW reservation state machine

How a job acquires and releases a scarce test bench. Prevents two jobs from clobbering the same hardware and emits telemetry for capacity planning.

stateDiagram-v2
  [*] --> Requested
  Requested --> Queued: bench busy
  Requested --> Acquired: bench free
  Queued --> Acquired: bench released
  Queued --> Expired: timeout (configurable)
  Acquired --> Running: tests start
  Running --> Released: tests complete
  Running --> Faulted: bench HW error
  Faulted --> Released: cleanup
  Released --> [*]
  Expired --> [*]

The reservation service exposes hwres acquire --bench=... and hwres release commands that workflows call. State transitions emit events to InfluxDB so utilization, queue depth, and fault rates are observable in Grafana.

Resource governance — preventing misuse

Six independent layers prevent runners from being used for non-CI/CD compute (the "crypto miner on the build farm" risk):

  1. Ephemeral pod runners — no persistent state survives a job.
  2. Hardened runner image — only approved toolchains; no apt install at runtime; signed with cosign.
  3. NetworkPolicy on runner namespaces — egress allowlist only to GHES, internal registry, MinIO, and approved services. Public internet blocked.
  4. OPA Gatekeeper admission control — denies pods escaping the runner profile.
  5. Required workflows — every run starts with a policy job that fails the run on anti-patterns (long sleep, suspicious binaries).
  6. Chargeback dashboards in Grafana — per-team visibility drives self-regulation.

Migration plan

Staged rollout over ~6–9 months. Do not big-bang.

Phase 0 — Foundation

4–6 weeks. Stand up GHES, ARC, CI pool for one pilot team, runner image pipeline, MinIO buckets, Grafana dashboards, reusable workflows library.

Phase 1 — CI parity

6–8 weeks. Migrate one product line's CI. Decompose Bamboo Python scripts into transparent reusable workflows. Run both in parallel for two release cycles.

Phase 2 — CD + Orchestrator

8–10 weeks. Stand up Release Orchestrator GitHub App, QGS, HW Pool. Migrate one product line's release flow. Keep the Postgres schema; point the new Orchestrator at it.

Phase 3 — Scale-out

Rolling. Onboard remaining teams one at a time, each cutover gated by their custom test systems being wired into QGS.

Phase 4 — Decommission

Retire Bamboo and the Spring Boot app once the last product line has moved.
